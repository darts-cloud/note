# ユースケース図構成仕様およびユースケース記述

ETロボコン2026 アプライドクラスの難所「ETラリー」攻略における、走行体と無線通信デバイスを包含したユースケース図の構成仕様（要素・関係）と、CockburnのSea level（ユーザー目標レベル）に準拠したユースケース記述を定義します。

---

## 1. ユースケース図構成仕様（UML記述仕様）

UML図そのものは描画せず、描くべき要素と関係性を表形式で定義します。人間はこの仕様をインプットとしてユースケース図を厳密に作図します。

### 1.1. システム境界とアクターの定義
* **システム境界**: `SOROT2026連携システム` (走行体と無線通信デバイスの協調全体)
* **アクター一覧**:

| アクター名 | 役割・分類 | 概要 |
|---|---|---|
| **走行体** | 主アクター (Active Actor) | 物理的にコースを走行し、センサやカメラを用いて現場情報の取得とアクチュエータ制御を担当する走行体。 |
| **無線通信デバイス** | 支援アクター (Supporting Actor) | 操作卓に固定され、Bluetooth通信を介して暗号の復号および高度な経路計算（探索）を支援する演算PC。 |
| **スターター** | 支援アクター (Supporting Actor) | キャリブレーション時に操作卓から4桁の復号キーを入力する人間。 |

### 1.2. ユースケース定義 (Sea levelPolicy)
CockburnのSea level（ユーザー目標）を満たす粒度で定義し、「画像を読み取る」「送信する」「復号する」といった手段・部分機能（Fish level）はユースケースとして抽出せず、ユースケース記述内のアクションフローとして記述します。

* **UC-201: ヒント情報を獲得する**: ゲートが設置されている全3つの座標をすべて特定し、走行準備状態にする。
* **UC-202: ゲート通過経路を計画する**: 特定したゲート座標を基に、最短かつ衝突しない安全なゲート通過ルートを生成する。
* **UC-203: ゲートを順次通過する**: 計画されたルートを自己位置補正を行いながら赤→青→黄の順に走行し、周回を重ねる。

### 1.3. 関連および関係性の一覧

UML表記上の混雑を避け、可読性を最大化するため、ユースケース間での無駄な `<<include>>` / `<<extend>>` の多用を避け、並列なフローとして定義します。

| 送信元要素 (From) | 関連種別 | 送信先要素 (To) | 関連端名 (ロール) | 多重度 | 意味・設計意図 |
|---|---|---|---|---|---|
| **走行体** | 関連 (Association) | **UC-201** | 探索・読取主体 | `1` : `1` | 走行体がカメラとオドメトリを用いてヒントカードを検出・走査する。 |
| **無線通信デバイス** | 関連 (Association) | **UC-201** | 復号処理担当 | `1` : `1` | 走行体から送られた暗号データをAES復号する。 |
| **スターター** | 関連 (Association) | **UC-201** | キー入力主体 | `1` : `1` | 事前にキーを入力し、復号に必要な情報を提供する。 |
| **走行体** | 関連 (Association) | **UC-202** | 計画リクエスト | `1` : `1` | 特定した座標を送り、経路の適用を待つ。 |
| **無線通信デバイス** | 関連 (Association) | **UC-202** | 経路計算担当 | `1` : `1` | マップ情報と座標から最適なノード移動シーケンスを計画する。 |
| **走行体** | 関連 (Association) | **UC-203** | 走行制御主体 | `1` : `1` | 計画された経路上を、自己位置を補正しつつ周回走行する。 |

## 2. ユースケース記述

抽出した3つの主要ユースケースについて、一連のアクションシーケンスを定義します。

### UC-201: ヒント情報を獲得する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-201: ヒント情報を獲得する** (上位要求: `REQ-401`, `REQ-402`) |
| **アクター** | 走行体 (主), 無線通信デバイス (支援), スターター (支援) |
| **事前条件** | 1. 競技スタート前のキャリブレーション時に、スターターにより無線通信デバイスに4桁の復号キー（`C-103`）が手入力されていること。<br>2. 走行体と無線通信デバイス間のBluetooth接続が確立していること (`C-105`)。<br>3. 走行体が競技スタートし、ヒントエリアの手前に到達していること。 |
| **事後条件** | 走行体のメモリ上に、赤・青・黄すべてのゲートが設置されている絶対座標（`C-102`）が格納され、経路計画可能な状態になっていること。 |
| **基本フロー** | **1. [走行体]** 搭載カメラを用いて自コース上の「ヒントカード1」を画像検出する。<br>**2. [走行体]** ヒントカード1の二次元コードを読み取り、平文の赤ゲート座標（例：`25,35`）を取得・記憶する (`REQ-401`)。<br>**3. [走行体]** 続いて自コース上の「ヒントカード2」を画像検出・デコードし、Base64の暗号文字列を取得する。<br>**4. [走行体]** Bluetooth通信を介して、暗号文字列を「無線通信デバイス」へ送信する。<br>**5. [無線通信デバイス]** 受信した暗号文をBase64デコードし、保持している復号キーを用いて **AES-128 (ECB)** で復号する (`REQ-402`, `C-103`)。<br>**6. [無線通信デバイス]** 復号された平文の青および黄ゲート座標（例：`53,54/12,22`）を抽出し、Bluetooth通信で「走行体」に返送する。<br>**7. [走行体]** 返送された座標データを受信し、赤・青・黄すべての位置を確定・記憶する。 |
| **代替フロー** | **[4a. Bluetooth通信エラー時 (FM-101)]**<br>1. 走行体は送信タイムアウトを検知し、安全にその場で一時停止する。<br>2. 走行体および無線通信デバイスはBluetoothの再接続シーケンスを自動実行する。<br>3. 再接続が確立した場合、基本フローの4番から処理を再開する。<br>**[3a. ヒントカード画像認識失敗時 (FM-102)]**<br>1. 走行体は二次元コードのデコード失敗を検知した場合、微速前進または後退をしてカメラのフォーカスと位置を調整する（リトライ）。<br>2. 規定回数リトライしても認識できない場合、走行体はオドメトリによる推定位置を信じて「ゲートが初期候補座標にある」と仮定して次に進む（フォールバック動作）。 |

### UC-202: ゲート通過経路を計画する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-202: ゲート通過経路を計画する** (上位要求: `REQ-403`) |
| **アクター** | 走行体 (主), 無線通信デバイス (支援) |
| **事前条件** | `UC-201`が正常に完了し、全ゲートの座標が特定できていること。 |
| **事後条件** | 走行体が赤→青→黄の順に通過し、他のゲートポジション（柱などの障害物）を回避するための、進入角・旋回Rを含んだ走行ルートノード（経路マップ）が走行体に設定されていること。 |
| **基本フロー** | **1. [走行体]** 特定した全ゲート座標情報を無線通信デバイスに送信し、最適経路の計算を要求する。<br>**2. [無線通信デバイス]** ゲートの向き（赤：横、青：縦、黄：横）および通過順序の制約（`C-101`）と、ゲートポジションGX-Yのグリッドマップ（障害物情報）を照合する。<br>**3. [無線通信デバイス]** 衝突のない最短通過ルート（進入方向および旋回ノード）を計算し、走行経路コマンド列を生成する (`REQ-403`)。<br>**4. [無線通信デバイス]** 生成した経路コマンドデータを走行体へ送信する。<br>**5. [走行体]** 経路データを受信し、走行用のコマンドキューに登録して走行準備を完了する。 |
| **代替フロー** | **[3a. 経路計算エラー（解なし）時]**<br>1. 無線通信デバイスは、なんらかの理由で規定順序（赤→青→黄）を通過する衝突なしルートが見つからない場合、安全マージンを削った緊急ルート（ゲート接触の可能性はあるが脱出できるルート）を計算し、警告フラグ付きで送信する。 |

### UC-203: ゲートを順次通過する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-203: ゲートを順次通過する** (上位要求: `REQ-404`, `REQ-405`) |
| **アクター** | 走行体 (主) |
| **事前条件** | `UC-202`が完了し、走行コマンドキューが作成されていること。 |
| **事後条件** | 制限時間内に、赤→青→黄の通過を最大3周回実行し、課題ポイントを獲得すること (`REQ-101`)。 |
| **基本フロー** | **1. [走行体]** 経路計画コマンドに基づき、オドメトリ制御（車輪エンコーダ・ジャイロ）を用いて赤ゲートへ向け移動を開始する。<br>**2. [走行体]** コース上に印刷された「ゲート位置補助情報」の二次元コードを検出・読み取る (`REQ-405`, `C-102`)。<br>**3. [走行体]** 読み取った補助情報座標（例：`B2`）と自身のオドメトリ自己位置推定値との偏差を算出し、位置情報（X, Y, 旋回角度）を補正（キャリブレーション）する。<br>**4. [走行体]** 赤色ゲート、青色ゲート、黄色ゲートを順番に走行体全体が通過する (`REQ-404`, `C-101`)。<br>**5. [走行体]** 黄色ゲート通過後、1周回成立フラグを立ててオドメトリ上の位置情報をリセットする。<br>**6. [走行体]** 再び1周目の赤ゲートに向けた経路コマンドに遷移し、規定時間内に最大3周回を繰り返し実行する。 |
| **代替フロー** | **[4a. 走行中に他のゲート（障害物）に接触またはコースを外れた場合]**<br>1. 走行体はジャイロセンサの急激な姿勢変化またはモータの過電流検知により「衝突・スタック」を判断する。<br>2. 走行体はその場でモータへの出力をゼロにし、安全に即時停止（競技終了宣言 `C-104`）を行い、暴走を防止する。 |
