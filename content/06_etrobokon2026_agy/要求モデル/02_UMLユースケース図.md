# ユースケース図構成仕様およびユースケース記述

ETロボコン2026 アプライドクラスの難所「ETラリー」攻略における、UMLユースケース図の構成仕様（システム境界、アクター、関連）と、プログラムによる完全自動走行を前提として主客関係を厳密化したユースケース記述を定義します。

---

## 1. ユースケース図構成仕様（UML記述仕様）

UMLの厳密な定義（アクター＝システムの外部からシステムを利用・支援するもの）に則り、主アクターを「競技者」に統一します。走行体および無線通信デバイスはシステム境界の内部要素（コンポーネント）として隠蔽し、競技中の人間による操作・起動表現を完全に排除して、プログラムによる自律自動走行モデルとして定義します。

### 1.1. System Boundary and Actors
* **システム境界**: `SOROT2026連携システム` (走行体ハードウェア、無線通信デバイスハードウェア、および各制御・計算ソフトウェアを含む全体)
* **アクター一覧**:

| アクター名 | 役割・分類 | 概要 |
|---|---|---|
| **競技者** | 主アクター (Primary Actor) | SOROT2026のメンバー（開発者・オペレーター・スターター）。競技開始前の「復号キー入力」および「スタート指示（スイッチ押下）」を行い、システムによる自動走行・周回走破という目標を達成する主体（競技走行中の介入・操作は一切行わない）。 |

> [!NOTE]
> 従来アクターとして扱っていた **「走行体」** および **「無線通信デバイス」** は、本システム境界の内部で動作する **構成要素（内部コンポーネント）** であるため、UMLの定義に従いアクター（外部要素）から排除し、図の可読性を最大化しています。また、復号キーを入力する「スターター」の役割も外部の人間として「競技者」に内包・統合しています。

### 1.2. ユースケース定義 (Sea levelPolicy - 完全自動自律)
CockburnのSea level（競技者から見たユーザー目標レベル）を満たす粒度で定義します。人間による「起動」や「操作」の能動記述を排除し、プログラムによる自律実行モデルとします。また、ヒントカードの読み取りおよび事前復号は必須ではないため、オプショナル（任意）機能であることを明確にします。

* **UC-201: [オプション] ゲート位置を特定する**: システムがヒントカードを読み取り・復号してゲートの設置絶対座標を特定するオプショナル機能（必須ではない）。
* **UC-202: [オプション] ゲート通過経路を計画する**: 特定した座標に基づき、システムが衝突のない安全なゲート通過ルートを自動的に生成・ロードするオプショナル機能（必須ではない）。
* **UC-203: ゲートを順次通過する**: 競技者のスタート指示を契機に、システムが自己位置補正を自動実行しつつ赤→青→黄の順にゲートを通過し周回を重ねる（ヒント特定が行われなかった場合は、リアルタイムなゲート色認識により自律走行を行う）。

### 1.3. 関連および関係性の一覧
主アクターである「競技者」から各ユースケースへの直接的な関連のみを定義します。競技者が行うアクションは、動作のトリガー（引き金）に厳格に限定されます。

| 送信元要素 (From) | 関連種別 | 送信先要素 (To) | 関連端名 (ロール) | 多重度 | 意味・設計意図 |
|---|---|---|---|---|---|
| **競技者** | 関連 (Association) | **UC-201** | キー入力・準備完了者 | `1` : `1` | 競技者がキャリブレーション時に復号キーを入力し、システムによるヒント特定準備を完了させる。 |
| **競技者** | 関連 (Association) | **UC-202** | 経路計画トリガー | `1` : `1` | システムが獲得した座標に基づき、自動で最適経路の計画とコマンドロードを実行する。 |
| **競技者** | 関連 (Association) | **UC-203** | スタート指示者 | `1` : `1` | 競技者がスタートスイッチを押すことで、システムが自律的にラリー走行および自動周回走破を開始・実行する。 |

## 2. ユースケース記述

### UC-201: [オプション] ゲート位置を特定する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-201: [オプション] ゲート位置を特定する** (上位要求: `REQ-401`, `REQ-402`) |
| **アクター** | 競技者 (主) |
| **位置づけ・概要** | **オプショナル（任意）機能**。計画走行を行う場合のみ自動実行されるユースケース。ヒントカードの読み取りおよび特定は必須ではなく、失敗時やスキップ時は自律的カラー追従走行で代替される。 |
| **事前条件** | 走行体と無線通信デバイス間のBluetooth接続が確立していること (`C-105`)。 |
| **事後条件** | システム（走行体側）のメモリ上に、赤・青・黄すべてのゲートが設置されている絶対座標（`C-102`）が格納され、経路計画可能な状態になっていること。※ 獲得に失敗した場合は、システムにヒント獲得失敗フラグが立ち、本ユースケースを終了する。 |
| **基本フロー** | **1. [競技者]** 競技スタート前のキャリブレーション時に、無線通信デバイス（システム）の画面から4桁の復号キー（`C-103`）を手入力する。<br>**2. [システム(デバイス)]** 入力された復号キーをオンメモリに記憶し、待機状態となる。<br>**3. [競技者]** 走行体のスタート指示を行う（スタートスイッチ押下）。<br>**4. [システム(走行体)]** 走行体が自動走行を開始し、ヒントエリアへ前進して搭載カメラを用いて「ヒントカード1」を自動検出する。<br>**5. [システム(走行体)]** ヒントカード1の二次元コードを読み取り、平文の赤ゲート座標（例：`25,35`）を抽出・記憶する (`REQ-401`)。<br>**6. [システム(走行体)]** 続いて「ヒントカード2」を自動検出し、Base64の暗号文字列を取得して、Bluetooth通信で無線通信デバイス（システム）へ自動送信する。<br>**7. [システム(デバイス)]** 受信した暗号文をBase64デコードし、登録された復号キーで **AES-128 (ECB)** 復号を行う (`REQ-402`, `C-103`)。<br>**8. [システム(デバイス)]** 復号された平文から青・黄ゲート座標（例：`53,54/12,22`）を特定し、走行体（システム）に自動返送する。<br>**9. [システム(走行体)]** 返送された座標データを受信し、全3ゲートの座標を自動で確定・記憶する。 |
| **代替フロー** | **[6a. Bluetooth通信エラー時 (FM-101)]**<br>1. システム(走行体)は送信タイムアウトを検知し、安全のためその場で自動一時停止する。<br>2. システムは再接続シーケンスを自動実行し、3.0秒以内に再接続に成功した場合はフロー6から処理を自動再開する。<br>3. 3.0秒以内に復旧しない場合、システムは「ヒント獲得失敗フラグ」を自動で立て、**「UC-203の自律的カラー追従走行モード（REQ-406）」** へ自動フォールバックする。<br>**[5a. ヒントカード画像認識失敗時 (FM-102)]**<br>1. システム(走行体)は画像認識・デコードエラーを検知した場合、微速動作による位置調整とカメラ露出自動補正をリトライする（3回）。<br>2. リトライ失敗時, システムは「ヒント獲得失敗フラグ」を自動で立て、**「UC-203の自律的カラー追従走行モード（REQ-406）」** へ自動的にフォールバックする。 |

### UC-202: [オプション] ゲート通過経路を計画する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-202: [オプション] ゲート通過経路を計画する** (上位要求: `REQ-403`) |
| **アクター** | 競技者 (主) |
| **位置づけ・概要** | **オプショナル（任意）機能**。特定したゲート座標を基に、最短かつ安全な走行ノードを自動計画する。ヒント特定が行われなかった場合は本ユースケースはスキップされる。 |
| **事前条件** | `UC-201`が正常に完了し、システム（走行体）側に全ゲートの座標が特定できていること。※ 特定できていない場合（ヒント獲得失敗フラグON）は、本ユースケースを自動的にスキップし、直接 `UC-203`（カラー追従自律走行モード）へ遷移する。 |
| **事後条件** | システム（走行体）側に、進入角や障害物回避ルートを含んだ走行コマンド列（経路マップ）が自動設定されていること。 |
| **基本フロー** | **1. [システム(走行体)]** 特定した全ゲート座標情報を無線通信デバイス（システム）に自動送信する。<br>**2. [システム(デバイス)]** ゲートの向き（赤：横、青：縦、黄：横）および通過順序の制約（`C-101`）とグリッドマップ（障害物情報）を照合する。<br>**3. [システム(デバイス)]** 最短かつ衝突のないゲート通過ルート（旋回ノードと進入方向）を自動計算し、走行コマンド列を生成する (`REQ-403`)。<br>**4. [システム(デバイス)]** 生成した経路データを走行体（システム）へ自動送信する。<br>**5. [システム(走行体)]** 経路データを受信し、走行用のコマンドキューに登録して走行準備を自動完了する。 |
| **代替フロー** | **[3a. 経路計画エラー（解なし）時]**<br>1. システム(デバイス)は衝突なしルートが計算できない場合、安全マージンを縮小した緊急エスケープルートを自動計算し、警告フラグ付きで走行体（システム）に自動送信し、適用する。 |

### UC-203: ゲートを順次通過する

| 項目 | 内容 |
|---|---|
| **ユースケースID / 名** | **UC-203: ゲートを順次通過する** (上位要求: `REQ-404`, `REQ-405`, `REQ-406`) |
| **アクター** | 競技者 (主) |
| **事前条件** | なし（事前計画走行用のコマンドがロードされている場合はそれを使用し、ロードされていない、またはエラーの場合は、自律的なカラー追従走行を自動実行する）。 |
| **事後条件** | 制限時間内に、赤→青→黄の順にゲートを最大3周回自動通過させ、課題ポイントを獲得すること (`REQ-101`)。 |
| **基本フロー** | **(事前計画走行モード時 - 競技者推奨ルート)**<br>**1. [競技者]** 走行体のスタート指示（スタートスイッチ押下）を行う。<br>**2. [システム(走行体)]** 計画された経路コマンドに基づき、オドメトリ制御とジャイロを用いて自律自動走行を開始する。<br>**3. [システム(走行体)]** コース上に印刷された「ゲート位置補助情報」の二次元コードを自動検出し、読み取り (`REQ-405`, `C-102`)、自身のオドメトリ自己位置推定値の累積誤差を自動補正する。<br>**4. [システム(走行体)]** 計画ルートに従い、赤色ゲート → 青色ゲート → 黄色ゲートを順番に自動通過する (`REQ-404`, `C-101`)。<br>**5. [システム(走行体)]** 黄色ゲート通過後、1周回成立フラグを立ててオドメトリ位置情報を自動リセットする。<br>**6. [システム(走行体)]** 再び赤ゲートに向けた経路コマンドに自動遷移し、規定時間内に最大3周回を繰り返し自動実行する。 |
| **代替フロー** | **[1a. 自律的カラー追従走行モード（ヒント復号なし）時 (REQ-406)]**<br>1. **[システム(走行体)]** ヒント獲得失敗時等において、自律的カラー追従走行モードへ自動で動作を切り替える（競技者の介入操作はなし）。<br>2. **[システム(走行体)]** 床面のメイン黒ラインを検出して緩やかに前進探索を開始する。<br>3. **[システム(走行体)]** 搭載カメラのカラー画像をリアルタイムに自動解析し、RGB/HSV色認識処理によってゲートの色（赤・青・黄）を識別・重心抽出する。<br>4. **[システム(走行体)]** 規定の通過順序（**赤 → 青 → 黄**）に適合する色のゲートをリアルタイムで自動検出し、カラー誘導制御（重心偏差に基づく操舵）を用いて自律的にゲートを通過する。<br>5. **[システム(走行体)]** 通過検知後、ターゲット色を次の色に切り替え（例：赤通過後は青）、探索と進入・走破を最大3周回繰り返し自動実行する。<br>**[4a. 走行中に他のゲート（障害物）に接触またはコースを外れた場合]**<br>1. **[システム(走行体)]** ジャイロ姿勢の急変やモータ過電流により「スタック・衝突」を自動検知する。<br>2. **[システム(走行体)]** 安全のためモータへの出力を即座にゼロにして自動緊急停止し、暴走を防止する（競技者による走行体回収・競技終了宣言 `C-104` の受理状態へ移行）。 |
