# 文脈エンジン 運用ガイド

> 参考動画：空気デザイン / Rio（YouTube）

---

## なぜ文脈エンジンが必要か

AIに毎回同じ文脈を説明し直すコストを「**コンテキストの再説明コスト**」と呼ぶ。

- 毎回、役職・プロジェクト状況・立場・期待する出力を説明しなければならない
- プロンプトエンジニアリングはこの根本問題を解決していない（対症療法）
- ChatGPT・Claude・Geminiなど各プラットフォームは意図的に文脈を囲い込んでいる（ウォールドガーデン）→ 乗り換えコストを上げるビジネスモデル

**本質的な問いかけ：** どのAIを使っても、説明ゼロで自分の文脈を完全に理解させるにはどうすればいいか？

---

## 解決策：文脈エンジン（Open Brain）

特定プラットフォームに依存しない、自分で所有する文脈データベース。
Nate B. Jones が提唱したコンセプト（PostgreSQL + MCP で実装）を、**Obsidian でコードなし・コストゼロ**で実現する。

---

## 理想の文脈エンジンが満たすべき4条件

| 条件                | 内容                                    |
| ----------------- | ------------------------------------- |
| 1. どのAIからもアクセスできる | Claude・Gemini・ChatGPT どれでも同じ文脈から始められる |
| 2. 自動的に最新状態を保てる   | 手動整理に頼らない。情報が自動で流れ込む設計                |
| 3. 整理された情報が蓄積される  | 生の情報のまま渡さない。重複・矛盾・ノイズを除去してからAIに渡す     |
| 4. AIへの目次がある      | AIがどのファイルを読むべきか迷わない。不要な情報を読ませない       |

> 条件4が最も見落とされやすい。目次なしで大量の情報を渡すとコンテキストウィンドウを無駄遣いし、ハルシネーションの原因になる。

## Obsidian ディレクトリ構造

```
/
├── 00_Inbox/          # 全ての情報の受け口
├── 01_Projects/       # 進行中のプロジェクト
├── 02_Ideas/          # アイデア・調査内容
├── 03_Resources/      # 参照した生の情報（第一次資料）
└── 04_Context/        # AI 向けの整理済み文脈ファイル群
    ├── MyContext.md         ← 目次 & AI アライバルプロトコル
    ├── AI-Handoff.md        ← AI 間の引き継ぎログ
    ├── ProfessionalIdentity.md
    ├── CurrentProjects.md
    ├── PhilosophyValues.md
    ├── VoiceMessaging.md    ← 話し方・トーンの特徴
    ├── WorkingStyle.md
    └── TechnicalSetup.md   ← 使用機材・ソフトウェア一覧
```

---

## MyContext.md（目次ファイル）の役割

- **AI アライバルプロトコル**として機能する
- Obsidian を訪れた AI はまずこのファイルを読む
- タスクに応じてどのファイルを読むべきかを案内する
- AI 間の引き継ぎ書（AI-Handoff.md）への参照も記載

---

## 情報の取り込み方法

| ソース        | 方法                                                            |
| ---------- | ------------------------------------------------------------- |
| ウェブページ     | Obsidian 公式ブラウザ拡張でワンクリック → Inbox に保存                          |
| YouTube 動画 | カスタム拡張（YT Obsidian）→ Mac Whisper で文字起こし → MD エクスポート → Inbox へ |
| スマートフォン    | Quick Capture Obsidian アプリで即座にメモ → Inbox に同期                  |
| AI との会話    | AIが学んだ内容を自動的に MyContext に追記                                   |

## 自動化の仕組み（Open Claw / ローカル AI）

```
毎朝 3〜4 時（自動実行）
  Inbox を読み込む
    ↓
  AI が分析・精査
    ↓
  Projects / Ideas / Resources / Context に振り分け
    ↓
  処理済みノートは Processed/ にアーカイブ
```

- 普段の処理：ローカル LLM（Open Claw の OSCAR）が無料で担当
- 週 1 回の大掃除：Claude Code でスケジューラー実行（壊れたリンク・重複・ゴミデータの削除）

---

## AI との接続方法（MCP サーバー）

Claude の場合：`~/Library/Application Support/Claude/claude_desktop_config.json` に数行追記するだけ。

```json
{
  "mcpServers": {
    "obsidian": {
      "command": "...",
      "args": ["<Obsidian vault のパス>"]
    }
  }
}
```

Open Claw の場合：`obsidian-cli` スキルを使用。

設定コードは Claude・Gemini などのチャットボットに生成してもらえばよい。

---

## ノート設計の秘訣：細かく分割する

- **一つのノートに大量の情報を詰め込まない**
- AI はノートを丸ごと読むため、不要な情報まで読み込んでしまう
- コンテキストウィンドウの節約とハルシネーション抑制のために、ファイルを細かく分割する
- ファイルが長くなりそうなら新しいノートを生成し、短く保つ
- 目次（MyContext）で「このタスクにはこのファイルを読め」と案内する

---

## 初期構築の手順

1. **既存データの移行**：Apple Notes・Notion などのデータを Obsidian インポータープラグインで移行
2. **AIに文書化を依頼**：メインの AI チャットボットに「私について知っていることをすべてノートにまとめて」と依頼し、MyContext の初版を作成
3. **MCP サーバーを設定**：使用する AI ツールと Obsidian を接続
4. **自動処理スクリプトを設定**：ローカル AI による Inbox 処理を自動化

---

## この仕組みの本質的な価値

> 「スキルの格差ではなく、インフラの格差」

- セットアップは数時間。本当の価値は **半年後・1年後**に現れる
- 毎回ゼロから説明する人 vs. 6ヶ月分の思考・意思決定が蓄積された人では、同じモデルを使っても出力の質が全く異なる
- 特定のプラットフォームに閉じ込められない
- 会社が変わっても消えない
- どの AI ツールを使っても持ち運べる文脈の資産
