## 重要な前提（規約を読まなくても知っておくこと）

### 採点・走行
- 走行ポイント = `max(35 − 走行タイム秒, 0)`、最大計測時間 120秒
- ETラリー課題ポイント: 1周回 5pt / 2周回 10pt / 3周回 15pt
- フライングスタート: **-35pt**（走行ポイント全損リスク）
- ゲート通過順は **赤→青→黄 固定**

### ETラリー特有
- ヒントカード1（赤・平文 `XY,XY`）: 1番目のゲート位置
- ヒントカード2（青/黄・**AES-128 ECB暗号 / Base64**）: 2-3番目のゲート位置
- 復号キー: テキスト4桁。**キャリブレーション中(1分以内)に操作台へ入力**（競技中は触れない）
- ゲート位置補助情報: コース上の5cm四方QRコードで自己位置補正可能

### 審査での重大事項
- 「**走行体のみで実現するモデルは評価が低くなる**」(規約 §3-6) → **走行体+無線通信デバイスの連携モデリング必須**
- 品質は **複数側面** から検討必須（信頼性・効率性・保守性・移植性など）
- 全モデル間の **段階的・双方向トレーサビリティ** が審査基準
- モデルとソフトウェア実装は **機能・構造・振舞いで関連** していること（規約 §3.2）

### UML/SysML図の作成方針（重要）
- **UML/SysML の図そのものは人間が任意のツール（Cacoo / Draw.io / Astah / OmniGraffle 等）で描画する**
- **マークダウンファイル内に PlantUML コードブロックを書かない**（一度試したが不採用）
- 各図に対応する `.md` ファイル（`01_SysML要求図.md` / `02_UMLユースケース図.md` 等）は **「描くべき要素・関係性の構成仕様」** を表形式で記述する
- 図を更新する場合は `.md` の構成仕様を先に更新 → 人間が図を再描画 という流れ
- `.puml` ファイルも作らない

### クロスリファレンス
- 同ボルト内: Obsidian ウィキリンク `[[ファイル名]]`（拡張子不要）
- リポジトリ全体での横断参照は `[[02_idea/...]]` のように相対パスを含める

### Mermaid
- ワークフロー図は Mermaid で記載可（Quartzでもネイティブ表示）。詳細は書きすぎず概要のみ

## PDFの読み方（重要・環境依存）

`Reference/` 配下のPDFはClaude CodeのReadツールから直接開けない（パスにより `pdftoppm not installed` エラー、または PATH の問題）。代わりに **テキスト抽出** で対応する。

### 前提
`brew install poppler` 済み（`/opt/homebrew/bin/pdftoppm`, `/opt/homebrew/bin/pdftotext`）。

### 抽出コマンド（セッション開始時に1回実行）

```bash
mkdir -p /tmp/etrc_pdf_text && \
for f in /Users/naoki/note/content/05_etrobokon2026/Reference/ETRC2026_*.pdf; do
  name=$(basename "$f" .pdf)
  /opt/homebrew/bin/pdftotext -layout "$f" "/tmp/etrc_pdf_text/${name}.txt"
done
```

抽出後は `/tmp/etrc_pdf_text/*.txt` を Read ツールで読める。
- `ETRC2026_participation_rules_V1.0.txt` (664行) — 参加規約・クラス区分・参加費・スケジュール
- `ETRC2026_rules_basic_applied_1.0.0.txt` (1968行) — 競技規約・コース・採点・難所定義
- `ETRC2026_dev_shinsa_rules_V1.0.txt` (1274行) — 審査規約・配点・モデル別記述ガイド

セクション抽出例:

```bash
# アプライドクラス競技ルール（章7）
awk '/^7\. 【A】競技：アプライドクラス/,/^8\. 無線通信デバイス/' \
  /tmp/etrc_pdf_text/ETRC2026_rules_basic_applied_1.0.0.txt

# アプライドクラス審査内容（3-1〜3-8）
sed -n '650,1064p' /tmp/etrc_pdf_text/ETRC2026_dev_shinsa_rules_V1.0.txt
```

## 自分（Claude）への注意

- **PDF直読みは失敗する** → 必ず上記 `pdftotext` フローを通す
- **PlantUML / `.puml` ファイルは作らない** → UML/SysML図は人間が描く。`.md` には「構成仕様（要素・関係の一覧表）」を書く
- **モデル審査対象は ETラリー** → ET相撲・ボトルデリバリーの要求/分析/設計/制御は書かない（実装は競技で必要だがモデル提出範囲外）
- **走行体+無線通信デバイスの2システム連携** をモデル全段階で意識（要求段階から見えるように）
- ID一覧の一次管理元:
  - REQ-XXX → `要求モデル/01_目標.md`（目標・ステークホルダー要求）
  - UC-XXX → `要求モデル/02_UMLユースケース図.md`
  - QA-XXX → `要求モデル/06_品質特性.md`
  - C-XXX → `要求モデル/08_制約.md`
  - FM-XXX → `要求モデル/04_FMEA.md`
- 規約PDF章節を参照するときは出典（§番号）を併記する
- FMEA は信頼性 QA-REL-* の達成根拠として位置づける（制御モデルにも引き継ぐ）
- **2025 CS審査総評** からの学び（`/tmp/etrc_pdf_text/ETRC2025_*.txt`）:
  - 用語定義はアドバンスト CS 0/6 のみ取り組み → 差別化要因
  - 可読性で 6/6 中 5チーム減点 → 印刷時の文字サイズ最重要
  - 上位3チームでも「サブシステム設計・アーキテクチャ」「制御戦略・要素技術」「概要・全体トレサビ」に課題
  - アブストラクトページは「本文のモデル」「特徴・結果・効果が一読でわかる」が高評価
- **UC粒度ポリシー**: Cockburn の Sea level（user goal）に揃える。Fish level（「ログインする」「読み取る」等の手段）は **UC図に書かず** ユースケース記述の基本フローに含める。ETロボコン2025 CSワークショップ P60 の基本UC図に倣う
