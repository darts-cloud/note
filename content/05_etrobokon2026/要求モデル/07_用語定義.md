# 用語定義（ETラリー / アプライドクラス）

要求モデル以降の全モデル間で使う **ユビキタス言語** を定義する。
2025 CS大会の指摘: **アドバンスト CS 0/6 チーム** しか用語定義に取り組んでおらず、取り組めば差別化要因になる。

> 根拠: 2025 CSワークショップ「より良いモデルの勘所」P14「用語の定義」
> 関連: [[00_要求モデル構成]] / [[03_SysML要求図]] / [[09_コンテキスト図]]

## 2. 自チーム命名（SOROT2026 ユビキタス言語）

ETラリー攻略のため自チームで定義する用語。**モデル全体で必ず統一**して使う。

### 2.1 機能・モジュール

| 用語 | 意味 | 利用条件 |
|---|---|---|
| ゲート認識器 | ゲートの色と位置を把握するモジュール（実装手段は設計で決定） | 常時 |
| カード認識器 | ヒントカード1/2/補助情報を統一的に読み取るモジュール | 手段A or C採用時 |
| 復号器 | ヒントカード2 を AES-128 復号するモジュール | 手段A採用時 |
| 位置情報統合器 | 3つのゲート位置を一元管理するモジュール | 常時 |
| 経路プランナ | 取得したゲート位置から走行経路を計画 | 常時 |
| 自己位置推定器 | 補助情報読取値で自己位置を補正 | 常時 |
| ライントレーサ | 黒線を追従して走行する制御 | 常時 |
| ゲートカウンタ | 通過ゲートの色順を判定する | 常時 |

### 2.2 状態・モード

| 用語 | 意味 |
|---|---|
| 待機モード | スタート合図前の状態 |
| 認識モード | ゲート色・位置の把握を試行中 |
| 走行モード | ライン追従＋ゲート通過試行中 |
| 復帰モード | ライン外れ等から回復試行中 |
| 完走モード | LAPゲート通過後の状態 |

### 2.3 データ

| 用語 | 意味 | 形式 |
|---|---|---|
| ゲート配置情報 | 3ゲートの `GX-Y` 座標と色のペア | `[(red, X, Y), (blue, X, Y), (yellow, X, Y)]` |
| 自己位置 | 走行体のコース上座標 | `(X, Y, θ)` |
| 走行ログ | 競技中の状態遷移記録 | タイムスタンプ + イベント |

## 3. 動詞（操作・振る舞い）

モデル全体で揃える主な動詞。

| 動詞 | 用例 | 不適切な揺れ表現 |
|---|---|---|
| 読み取る | 「ヒントカードを読み取る」 | 取得する／認識する／取り込む |
| 復号する | 「ヒントカード2を復号する」 | デコードする／展開する |
| 走行する | 「ライン上を走行する」 | 移動する／進む |
| 通過する | 「ゲートを通過する」 | 通る／抜ける |
| 補正する | 「自己位置を補正する」 | 修正する／調整する |
| 計画する | 「走行経路を計画する」 | 設計する／決定する |

## 4. NG例（モデル中で混在を許さない）

2025 CS指摘事項より:

- ❌ 「青ライン(青線)」を 場面によって「青ライン」「青線」 と書き分け → ✅ どちらかに統一
- ❌ 「ヒントカード」「ヒントQR」「ヒント二次元コード」を混在 → ✅ 「ヒントカード」に統一
- ❌ 「無線通信デバイス」「操作台PC」「外部PC」を混在 → ✅ 「無線通信デバイス」（規約用語）に統一
