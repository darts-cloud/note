# アプライドクラス 要求モデル（A3横1枚）

ETロボコン2026 アプライドクラスのモデル審査における **「要求モデル」** （配点20%）に提出する1枚のドキュメント設計。

> 関連:
> - 上位: [[09_内容と図の選定]] - CS大会提出物全体（A3横6枚）の中での位置づけ
> - 規約: `Reference/ETRC2026_dev_shinsa_rules_V1.0.pdf` §3-2, §3-6 (要求モデル①②)
> - 参考: [[04_knowledge/etrobocon-team-guideline]]

---

## スコープ

| 項目 | 内容 |
|---|---|
| 対象クラス | アプライドクラス（HackSPi: SPIKE + Raspberry Pi） |
| 審査対象課題 | **ETラリー限定**（ET相撲・ボトルデリバリーは審査対象外） |
| 提出枚数 | 地区大会・CS大会ともに 1ページ（A3横） |
| 配点 | 20%（モデル4種・アブストラクト・総合の中で2番目に大きい） |
| 修正可能性 | 地区大会提出後、CS大会で改訂可 |

---

## 審査基準（規約 §3-4 要約）

| 観点 | 審査項目 | 我々の応答 |
|---|---|---|
| 妥当性 | 開発目標と、それを実現する機能・品質・制約が十分検討されているか | Layer 1（目標）〜Layer 4（品質・制約）を網羅 |
| 妥当性 | 品質の検討は複数側面からなされているか | [[06_品質特性]] で信頼性・効率性・保守性・移植性の4軸 |
| トレーサビリティ | 段階的かつ適切な分解により追跡可能か | 要求図で `<<contain>>` / `<<derive>>` / `<<refine>>` を明示 |

| #   | 構成要素（記載内容）                                                                              | 表現方法                                                                                                      | 詳細ファイル            |
| --- | --------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- | ----------------- |
| 1   | **開発目標と達成指標** — CS大会上位入賞という目的に対して、ETラリー攻略・規定時間内完走・走行ポイント獲得・モデル審査評価の4目標を客観的に測定可能な指標で表明する | 表（ID／目標／達成指標／派生戦略）＋戦略の自然言語記述                                                                              | [[01_目標]]         |
| 2   | **ステークホルダー要求** — 競技・審査委員会・チームの各視点で目標から派生する要求                                         | 表（ID／ステークホルダー／内容／派生元REQ）                                                                                  | [[01_目標]]         |
| 3   | **機能ユースケース** — ETラリー攻略に必要な機能を、アクターとシステム境界・UCグループで俯瞰する                                   | **UMLユースケース図**（アクター／システム境界／UCパッケージ／include・extend）                                                        | [[02_UMLユースケース図]] |
| 4   | **ユースケース詳細** — 各UCの正常系・代替系・例外系の振る舞いを規定する                                                | ユースケース記述テンプレート（概要／関連アクター／事前条件／事後条件／不変条件／トリガ／基本フロー／代替フロー／例外フロー／関連UC）                                       | [[03_ユースケース記述]]   |
| 5   | **信頼性根拠（故障モード）** — 各機能の起こりうる故障と影響を評価し、信頼性要求の達成根拠とする                                     | FMEA表（機能／故障モード／影響／原因／重大度S・発生度O・検出度D・RPN／対策／関連REQ・QA）＋ 評価尺度（1〜10）                                          | [[04_FMEA]]       |
| 6   | **要求トレーサビリティ** — 目標→ステークホルダー要求→機能要求→性能要求→品質特性／制約 の段階的分解を可視化する                           | **SysML要求図**（5層構造／`<<containment>>` `<<deriveReqt>>` `<<refine>>` `<<satisfy>>` `<<verify>>` `<<trace>>`） | [[05_SysML要求図]]   |

| 8   | **ユビキタス言語（用語定義）** — モデル全体（要求／分析／設計／制御）で揺れず使う共通語彙                                        | 用語定義表（規約由来 ＋ 自チーム命名／動詞統一）                                                                                 | [[07_用語定義]]       |

### 表現方法の選択ポリシー

| 種類 | 採用基準 |
|---|---|
| **UML/SysML図** | 階層・関係・並列構造を見せるとき（要求図・UC図） |
| **表** | 列挙・属性管理・指標値が中心のとき（REQ／QA／C／FM） |
| **自然言語** | 戦略の背景・補足情報・規約引用（50字以内が目安） |
| **数式** | 性能要求・採点ロジック（例: `走行ポイント = max(35 − t, 0)`） |

### 全体に共通する留意点

- **走行体＋無線通信デバイスの2システム連携** が要求段階から見えること（規約 §3-6 補足情報 システム分析: 走行体のみのモデルは評価が低くなる）
- **図そのもの（要求図・UC図）は人間が任意のツールで描画する**。各 `.md` ファイルは「何を描くか」の構成仕様
- **印刷時の文字サイズは10pt以上**（規約 クラス共通注意事項。2025 CSで 6/6 中 5チーム減点）
- **ダイアグラムヘッダー** を付ける（`req:` `uc:` `bdd:` `ibd:` 等。2025 ワークショップ推奨）

## 命名規則

| 種別 | プレフィックス | 例 |
|---|---|---|
| 要求 | `REQ-` | `REQ-401` |
| ユースケース | `UC-` | `UC-3` |
| 品質要求 | `QA-` | `QA-PERF-01` |
| 制約 | `C-` | `C-101` |
| FMEA項目 | `FM-` | `FM-001` |

ID重複・採番衝突に注意。新規追加は **追加先ファイルの一覧表** に必ず追記する。
要求IDは [[01_目標]]、UCは [[02_UMLユースケース図]]、QAは [[06_品質特性]]、制約は [[08_制約]]、FMは [[04_FMEA]] が一次管理元。

## 関係種別（SysML要求図）

| ステレオタイプ | 用途 |
|---|---|
| `<<containment>>` | 上位要求が下位要求を包含する |
| `<<deriveReqt>>` | ある要求から派生した要求 |
| `<<refine>>` | ユースケース等で要求を洗練する |
| `<<satisfy>>` | 設計要素が要求を満たす（システム分析モデル以降で使用） |
| `<<verify>>` | テストが要求を検証する（制御モデル以降で使用） |
| `<<trace>>` | 競技規約等のドキュメントとの追跡関係 |

詳細な使い分けは [[05_SysML要求図]] 参照。
