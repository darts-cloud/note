# ユースケース記述（ETラリー）

[[02_UMLユースケース図]] で定義した各 Sea level ユースケースの **詳細記述**。
規約 §3-2 表で要求モデル主要図に「ユースケース記述」が列挙されている。
2025 CSワークショップ「より良いモデルの勘所」P56-57 の様式に従う。

> 関連: [[02_UMLユースケース図]] / [[01_目標]] / [[04_FMEA]] / [[07_用語定義]]

---

## 1. 記述様式

各UCに対してフルバージョンで記述する。粒度は **分析レベル**（設計詳細には踏み込まない）。
**Fish level の手段（ヒントカード認識・復号・ライン走行 等）は基本フローの手順として記述する。**

| 項目 | 内容 |
|---|---|
| ユースケース名 | 「○○を○○する」述語形式 |
| 概要 | 1〜2文でこのUCの意義 |
| 関連アクター | 主アクター・副アクター |
| 事前条件 | UC開始前に真でなければならない条件 |
| 事後条件 | UC終了時に満足する条件（測定可能な達成基準） |
| 不変条件 | UC実行中に常に成立する条件（任意。「契約による設計」） |
| トリガ | UCを開始するイベント |
| 基本フロー | 正常系の手順（Fish level の手段はここに含める） |
| 代替フロー | 条件分岐がある場合の別パス |
| 例外フロー | 失敗時のリカバリ手順（FM-XXX とリンク） |
| 関連ユースケース | include/extend で繋がる他UC |

## 2. ユースケース記述

### UC-1 ETラリーで高得点を獲得する

| 項目 | 内容 |
|---|---|
| 概要 | スターターが走行体を運用し、ETラリー競技で課題ポイントと走行ポイントを最大化する |
| 関連アクター | スターター（主）、開発審査員（観察） |
| 事前条件 | 受付・車検が完了し、ターン開始指示が出ている |
| 事後条件 | リザルトポイント ≥ 35pt（走行ポイント+課題ポイントの合計） |
| 不変条件 | 競技規約を遵守している（フライング・コース逸脱・違反操作なし） |
| トリガ | 審判の「ターン開始」指示 |
| 基本フロー | 1. UC-2 競技準備を完了する<br>2. スタート合図に応じて走行体をスタートさせる<br>3. UC-3 ゲートの色と位置を把握する<br>4. UC-4 計画した経路でゲートを通過する（複数周回試行）<br>5. UC-5 ガレージへフィニッシュする |
| 代替フロー | 走行体が走行不能に陥った場合: 競技終了宣言を申し出る |
| 例外フロー | フライングスタート発生: -35pt 確定。残り時間で課題ポイントのみ獲得を狙う（FM-001） |
| 関連UC | UC-2, UC-3, UC-4, UC-5（include） |

### UC-2 競技準備を完了する

| 項目 | 内容 |
|---|---|
| 概要 | スターターがキャリブレーション時間（1分）内に走行体・無線通信デバイスを整え、スタート可能な状態にする |
| 関連アクター | スターター（主）、競技審判（副・指示） |
| 事前条件 | 審判のキャリブレーション開始指示が出ている |
| 事後条件 | 走行体がスタートエリア内に完全停止状態で設置され、復号キーが無線通信デバイスに保持されている |
| 不変条件 | キャリブレーション時間内（1分以内） |
| トリガ | 審判の「キャリブレーション開始」指示 |
| 基本フロー | 1. 無線通信デバイスを操作台に設置する<br>2. センサ調整（カラーセンサ・ジャイロ等）を実施する<br>3. *（UC-3 で手段A採用時）* 操作台に表示された復号キー4桁を無線通信デバイスへ入力する<br>4. 調整用ガジェットを操作台へ撤去する<br>5. 走行体をスタートエリアへ完全停止状態で設置する |
| 代替フロー | UC-3 で手段A（ヒント経由）を採用しない場合: 手順3 をスキップ |
| 例外フロー | 1分超過: 審判の判断でスタート準備へ強制移行（手段A採用時は FM-004） |
| 関連UC | UC-1（included from） |
| 備考 | 手順3（復号キー入力）は UC-3 の手段選択に依存する **保留要素**。手段A採用時は C-108 への対応として必要。手段B/C採用時は不要 |

### UC-3 ゲートの色と位置を把握する

| 項目 | 内容 |
|---|---|
| 概要 | 走行体が3つのゲート（赤・青・黄）それぞれの色と位置を把握し、経路計画に必要な情報をそろえる。**実現手段は抽象化**（ヒント経由・直接視覚認識・併用は設計モデルで決定） |
| 関連アクター | スターター（主・走行体経由）、開発審査員（観察） |
| 事前条件 | スタート合図発生済み |
| 事後条件 | 3ゲートそれぞれの (色, 位置) が走行体内部に保持されている |
| 不変条件 | 走行タイム ≤ 120秒 |
| トリガ | 走行開始 |
| 基本フロー | 1. 走行体がコース上を走行する<br>2. **ゲートの色と位置を把握する**（手段は設計モデルでトレードオフ分析）<br>3. 把握した3ゲートの (色, 位置) を経路プランナへ渡す |
| 代替フロー | 把握できないゲートがある場合: 把握済みゲートのみで部分的経路計画する |
| 例外フロー | 把握失敗: 警告ログ出力し走行継続。可能なら走行中に再試行（FM-002, FM-003） |
| 関連UC | UC-1（included from）、UC-4（このUCの結果を使用） |
| 備考 | **実現手段の候補**（設計モデルで選択）:<br>・**手段A** ヒント経由: ヒントカード1（平文）+ ヒントカード2（AES-128 ECB復号）<br>・**手段B** 直接認識: 走行中にカメラでゲート色と位置を視覚認識<br>・**手段C** 補助情報併用: ゲート位置補助情報（コース上QR）で自己位置補正<br>・**手段D** ハイブリッド: A+B 併用、失敗時フォールバック<br><br>手段A採用時は UC-2 で復号キーを保持し、**走行体↔無線通信デバイス連携** を使う |

### UC-4 計画した経路でゲートを通過する

| 項目 | 内容 |
|---|---|
| 概要 | 走行体が取得したゲート位置情報から走行経路を計画し、赤→青→黄の固定順でゲートを通過する。複数周回を試行して課題ポイントを最大化 |
| 関連アクター | スターター（主・走行体経由）、開発審査員（観察） |
| 事前条件 | UC-3 完了（最低1ゲート位置取得済み） |
| 事後条件 | 最低1周回（3ゲート通過）成立、可能なら3周回まで累積 |
| 不変条件 | 走行タイム ≤ 120秒、ライン上を走行している |
| トリガ | UC-3 から3ゲート位置情報受領 |
| 基本フロー | 1. ゲート位置から赤→青→黄の最短経路を **計画する**<br>2. ライン上を **走行する**（PID制御）<br>3. コース上のゲート位置補助情報（5cm四方の二次元コード）を読み取り、**自己位置を補正する**<br>4. ゲート位置に到達したら **ゲート色を判定し通過する**<br>5. 赤→青→黄 通過完了で **1周回成立**<br>6. 残時間があれば次周回を開始（手順 2〜5 繰り返し） |
| 代替フロー | UC-3 でゲート2,3が不明: 既知ゲート1のみで部分的経路を生成し、できる限り通過する |
| 例外フロー | ライン外れ: 復帰モードへ（FM-007）<br>ゲート色順違反: 周回未成立として再周回開始（FM-006）<br>自己位置補正失敗: 推定位置で継続走行（FM-005） |
| 関連UC | UC-1（included from）、UC-3（このUCの前段）、UC-5（このUCの後段） |
| 備考 | 走行体内部処理が中心。走行ログは **無線通信デバイス経由で記録** される（2システム連携） |

### UC-5 ガレージへフィニッシュする

| 項目 | 内容 |
|---|---|
| 概要 | 規定周回数達成または時間制限近接で走行体をガレージに導き完全停止させ、フィニッシュポイント3ptを獲得する |
| 関連アクター | スターター（主・走行体経由）、競技審判（フィニッシュ判定） |
| 事前条件 | UC-4 が1回以上成立 |
| 事後条件 | 走行体がガレージ内に完全停止し、フィニッシュ判定が成立 |
| 不変条件 | 走行タイム ≤ 120秒（最大計測時間内） |
| トリガ | 周回完了判断 or 時間制限警告 |
| 基本フロー | 1. ガレージ方向へ進路変更<br>2. ガレージ手前の黒線を検出<br>3. ガレージ内（灰色囲み白領域）に進入<br>4. 完全停止 |
| 代替フロー | LAPゲート到達後の継続走行（最大計測時間まで課題ポイント獲得を狙う） |
| 例外フロー | ガレージ手前で停止失敗: フィニッシュ未成立 -3pt（FM-009）<br>時間切れ: 走行ポイントは最大計測時間扱い |
| 関連UC | UC-1（included from）、UC-4（このUCの前段） |

### UC-6 走行結果を判定する

| 項目 | 内容 |
|---|---|
| 概要 | 競技審判がリザルトポイントを計算・提示し、参加チームの了承を得る |
| 関連アクター | 競技審判（主）、スターター（副・了承） |
| 事前条件 | UC-1 のターンが終了している（最大計測時間経過、LR双方フィニッシュ、または失格） |
| 事後条件 | リザルトポイントが確定し、参加チームが了承している |
| トリガ | ターン終了 |
| 基本フロー | 1. 走行ポイント `max(35 − 走行タイム, 0)` を計算<br>2. 課題ポイント（ETラリー周回数 × 5/10/15pt）を集計<br>3. リザルトポイント `max(走行+課題, 0)` を計算<br>4. 参加チームへ提示<br>5. 異議がなければ確定 |
| 代替フロー | 参加チームが異議申立て: 審判が再確認し、必要なら修正 |
| 例外フロー | 失格事由発生時: リザルトポイント0 確定 |
| 関連UC | UC-1（include） |

## 3. 記述上の注意点

- **「○○を○○する」述語形式** で名称統一（規約準拠）
- **Fish level の手段は基本フロー内に手順として記述**（UC として分離しない）
- **下位ユースケースを記述に含めない**: 1つ下のレベルまで言及。2レベル下以降は呼ばない（規約 §61）
- **用語は [[07_用語定義]] に従う**（揺れ禁止。例「読み取る」を「取得する」に書き換えない）
- **失敗シナリオは FMEA とリンク**（FM-XXX 番号で参照）
- **走行体↔無線通信デバイス連携を基本フローで明示**（規約 §3-6 への対応）
