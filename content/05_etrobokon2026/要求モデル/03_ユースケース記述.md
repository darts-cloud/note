# ユースケース記述（ETラリー）

[[02_UMLユースケース図]] で定義した各ユースケースの **詳細記述**。
規約 §3-2 表で要求モデル主要図に「ユースケース記述」が列挙されている。
2025 CSワークショップでは **ユースケース図単体ではなく記述併記** が推奨されている。

> 関連: [[02_UMLユースケース図]] / [[01_目標]] / [[04_FMEA]] / [[07_用語定義]]

---

## 1. 記述様式

各UCに対して以下のテンプレートで記述する（フルバージョン）。粒度は **分析レベル**（設計詳細には踏み込まない）。

| 項目       | 内容                           |
| -------- | ---------------------------- |
| ユースケース名  | 「○○を○○する」述語形式                |
| 概要       | 1〜2文でこのUCの意義                 |
| 関連アクター   | 主アクター・副アクター                  |
| 事前条件     | UC開始前に真でなければならない条件           |
| 事後条件     | UC終了時に満足する条件                 |
| 不変条件     | UC実行中に常に成立する条件（任意。「契約による設計」） |
| トリガ      | UCを開始するイベント                  |
| 基本フロー    | 正常系の手順                       |
| 代替フロー    | 条件分岐がある場合の別パス                |
| 例外フロー    | 失敗時のリカバリ手順                   |
| 関連ユースケース | include/extend で繋がる他UC       |

## 2. ユースケース記述

### UC-101 復号キーを入力する

| 項目 | 内容 |
|---|---|
| 概要 | スターターが操作台に表示された4桁の復号キーを無線通信デバイスに入力する |
| 関連アクター | スターター（主） |
| 事前条件 | キャリブレーション時間（1分）が始まっている |
| 事後条件 | 復号キーが無線通信デバイスに保持され、復号器から参照可能 |
| トリガ | 審判の「キャリブレーション開始」指示 |
| 基本フロー | 1. 操作台で復号キー4桁を確認<br>2. スターターが無線通信デバイスにキー入力<br>3. 確定操作で復号キーが保持される |
| 代替フロー | なし |
| 例外フロー | 1分以内に入力完了できない場合: 審判の判断でスタート準備へ移行（UC-203 が成立しない可能性） |
| 関連UC | UC-102（並行で実施）、UC-203（このUC完了が前提） |

### UC-102 キャリブレーションを実施する

| 項目 | 内容 |
|---|---|
| 概要 | スターターが走行体・無線通信デバイスを調整し、スタートエリアへ走行体を設置する |
| 関連アクター | スターター（主）、審判（副・指示） |
| 事前条件 | 審判のキャリブレーション開始指示が出ている |
| 事後条件 | 走行体がスタートエリア内に完全停止状態で設置され、調整用ガジェットが撤去されている |
| トリガ | 審判の「キャリブレーション開始」指示 |
| 基本フロー | 1. センサ調整実施<br>2. 調整用ガジェットを操作台へ撤去<br>3. 走行体をスタートエリアへ設置<br>4. 走行体を完全停止状態で待機 |
| 代替フロー | UC-101 と並行実施 |
| 例外フロー | 1分超過: 審判の判断でスタート準備へ強制移行 |
| 関連UC | UC-101 |

### UC-201 ヒントカード1を読み取る

| 項目 | 内容 |
|---|---|
| 概要 | 走行体のカード認識器でヒントカード1（赤・平文 `XY,XY` 形式）を読み取り、1番目のゲート位置を取得する |
| 関連アクター | （システム内部処理のため外部アクターなし） |
| 事前条件 | スタート合図後、走行体が稼働中である |
| 事後条件 | 1番目のゲート位置（`GX-Y`）が走行体内部に保持される |
| トリガ | 走行体がヒントカード1の読取位置に到達 |
| 基本フロー | 1. カメラ画像取得<br>2. QRデコード（平文）<br>3. `XY,XY` 文字列をパース<br>4. 1番目ゲート位置として保存 |
| 代替フロー | なし（ヒントカード1は平文のため復号不要） |
| 例外フロー | 認識失敗: リトライ（最大3回）→ 失敗時は警告ログ出力し走行継続（FM-002 参照） |
| 関連UC | UC-301（このUCの結果を統合） |

### UC-202 ヒントカード2を読み取る

| 項目 | 内容 |
|---|---|
| 概要 | 走行体のカード認識器でヒントカード2（青/黄・暗号 Base64）を読み取り、UC-203 へ渡す |
| 関連アクター | （内部処理） |
| 事前条件 | スタート合図後 |
| 事後条件 | 暗号文字列が復号器の入力に渡されている |
| トリガ | ヒントカード2読取位置到達 |
| 基本フロー | 1. カメラ画像取得 → 2. QRデコード（Base64文字列）→ 3. 復号器へ渡す |
| 例外フロー | 認識失敗: リトライ（最大3回）→ 失敗時は走行継続 |
| 関連UC | UC-203（include） |

### UC-203 ヒントカード2を復号する

| 項目 | 内容 |
|---|---|
| 概要 | 復号器が AES-128 ECB で暗号文を復号し、2-3番目のゲート位置を取得する |
| 関連アクター | （内部処理） |
| 事前条件 | UC-101 完了（復号キー保持済み）AND UC-202 完了（暗号文取得済み） |
| 事後条件 | 2番目・3番目のゲート位置が走行体内部に保持される |
| トリガ | UC-202 から復号入力受領 |
| 基本フロー | 1. Base64デコード<br>2. AES-128 ECB復号（鍵=UC-101のキー）<br>3. `XY,XY/XY,XY` パース<br>4. ゲート2, 3 位置を保存 |
| 例外フロー | 復号失敗（キー不正など）: ゲート2,3を不明状態にし、警告ログ出力（FM-003 参照） |
| 関連UC | UC-101（include）、UC-202（included from） |

### UC-301 ゲート位置情報を統合する

| 項目 | 内容 |
|---|---|
| 概要 | UC-201, UC-203 で得た3ゲートの位置情報を一元管理する |
| 関連アクター | 開発審査員（観察） |
| 事前条件 | UC-201 完了、UC-203 完了（または部分完了） |
| 事後条件 | 3ゲートの (色, X, Y) リストが経路プランナへ渡される |
| トリガ | UC-201/UC-203 から位置情報受領 |
| 基本フロー | 1. 位置情報を受領<br>2. (色, X, Y) リストに統合<br>3. 経路プランナへ通知 |
| 代替フロー | UC-203 失敗時: 既知のゲート1のみで経路計画する |
| 関連UC | UC-201, UC-203（include）、UC-302（このUCの後段） |

## 3. 記述上の注意点

- **「○○を○○する」述語形式** で名称統一（規約準拠）
- **下位ユースケースを記述に含めない**: 1つ下のレベルまで言及。2レベル下以降は呼ばない（規約 §61）
- **アクターと「内部処理」の区別**: 外部主体の場合のみアクター明記
- **用語は [[07_用語定義]] に従う**（揺れ禁止）
- **失敗シナリオは FMEA とリンク**（FM-XXX 番号で参照）
