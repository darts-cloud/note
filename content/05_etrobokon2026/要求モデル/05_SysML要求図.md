# SysML要求図 — 構成仕様

要求の階層構造とトレーサビリティを示す **要求図** の中身を仕様化する。
**図そのものは人間が任意のツール（Cacoo / Draw.io / Astah / OmniGraffle 等）で描画する** — このファイルは「何を描けばいいか」の入力資料。

> 関連: [[00_要求モデル構成]] / [[02_UMLユースケース図]] / [[06_品質特性]] / [[08_制約]] / [[04_FMEA]]
> 規約: 審査規約 §3-4 妥当性 / トレーサビリティ、§3-6 補足情報（要求モデル①②）

---

## 1. 図の目的

| 観点 | 内容 |
|---|---|
| 規約対応 | 「段階的かつ適切な分解により追跡可能になっているか？」(§3-4 トレーサビリティ) |
| 主な読み手 | 開発審査員（モデルの妥当性・トレース確認） |
| 表現対象 | 目標→ステークホルダー要求→システム要求→品質/制約 の階層と関係 |
| 配置 | A3横1枚の左上に最大面積で配置（[[09_内容と図の選定]] 参照） |

## 2. 階層構造（5層）

| Layer | パッケージ名 | 含むノード種別 | ID範囲 |
|---|---|---|---|
| 1 | 開発目標 | requirement | REQ-001〜004 |
| 2 | ステークホルダー要求 | requirement | REQ-101, REQ-201〜204, REQ-301 |
| 3 | 機能要求 | requirement | REQ-401〜409 |
| 3 | 性能要求 | requirement | REQ-501〜503 |
| 4 | 品質要求 | quality（4分類見出し） | QA-REL-*, QA-PERF-*, QA-MNT-*, QA-PORT-* |
| 4 | 制約 | constraint（3分類見出し） | C-1xx, C-2xx, C-3xx |

各 REQ の本文・指標は [[00_要求モデル構成]] の一覧表で管理。要求図ではノード見出しだけを表示する。

---

## 3. 関係線（トレーサビリティ）

矢印種別と意味:

| ステレオタイプ | 意味 | 使う場面 |
|---|---|---|
| `<<contain>>` | 上位が下位を包含 | Layer 1 → Layer 2 |
| `<<derive>>` | 上位から派生 | Layer 1/2 → Layer 3 |
| `<<refine>>` | 抽象を具体化 | Layer 3 → Layer 4、機能間の詳細化 |
| `<<satisfy>>` | 設計要素が要求を満たす | 後段モデル（システム分析以降）で使用 |
| `<<verify>>` | テストが要求を検証 | 後段モデル（制御モデル）で使用 |

## 5. 描画上の注意点（人間向け）

- **配置**: Layer 1 を上、下に行くほど詳細化（縦軸 = 抽象度）
- **色分け**: requirement=黄、quality=緑、constraint=赤 を推奨（[[09_内容と図の選定]] と統一）
- **凡例を必ず付ける**: ステレオタイプの意味を端に小さく載せる
- **REQ-IDは必ずノードに表示**: 後段モデルとのトレースに使われる
- **印刷時のフォントは10pt以上**（規約 §クラス共通 注意事項）
- **ノードに本文を全て書かない**: タイトル＋ID＋指標 程度に留め、本文は [[00_要求モデル構成]] 側に置く

- [ ] 関係線にステレオタイプが付いているか
