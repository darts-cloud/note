# 制約一覧（ETラリー / アプライドクラス）

要求モデルが満たすべき **制約** を整理する。規約由来・物理環境・開発の3区分で管理。
規約 §3-6 補足情報（要求モデル①）: 「システムを構築する上では前提となる各種の制約がある。こうした制約も重要な要求となる」

> 関連: [[00_要求モデル構成]] / [[05_SysML要求図]] / [[06_品質特性]]

---

## 凡例

| 列 | 内容 |
|---|---|
| ID | `C-<区分>-<連番>`。区分: 1xx=規約由来 / 2xx=物理環境 / 3xx=開発 |
| 制約内容 | 何が制約されるか |
| 出典 | 競技規約（基本・応用）／参加規約／審査規約のいずれか + 章節 |
| 関連要求 | この制約が紐づく `REQ-xxx` |

## 制約から派生する要注意点

1. **C-104（無線通信デバイス競技中操作禁止）** → 復号キー入力はキャリブレーション中(1分)に完了させる必要。UC-101 はキャリブレーション内に閉じ込める設計。
2. **C-107/C-108（AES-128 + 1分キャリブレーション）** → 復号キー入力UI と復号処理の検証を実機で必ず行う。シミュレータでは復号処理省略可だが、実機検証必須。
3. **C-202（シミュ↔実機差）** → 移植性 QA-PORT-01 を品質目標に置き、両環境でのテストを設計プロセスに組み込む。
4. **C-307/C-308** → モデル全体（要求/分析/設計/制御）はETラリー集中。他難所はソースコード上は実装しても、モデル提出物には含めない判断もあり得る。
5. **C-110（フライングスタート -35pt）** → スタート判定の信頼性は走行ポイント全損リスク。要素技術検討時に最優先で安定化。

## 改訂ルール

- 規約改訂（V1.0以降）があった場合、出典列を更新し変更履歴を残す
- 「（経験則）」「（内部）」と書かれた制約は試走後に根拠を追記
- 大会開催地告知時に C-203, C-204, C-205 に会場固有情報を追記
