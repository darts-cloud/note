# UMLユースケース図 — 構成仕様

ETラリー攻略システムの **ユースケース図** の中身を仕様化する。
このファイルは「何を描けばいいか」の入力資料。

> 関連: [[00_要求モデル構成]] / [[03_ユースケース記述]] / [[05_SysML要求図]]
> 規約: 審査規約 §3-2 表（要求モデル主要図）、§3-6 補足情報（システム分析）
> 参考: ETロボコン2025 CSワークショップ「より良いモデルの勘所」P60 基本のユースケース図

## 2. ユースケース粒度のポリシー

**Cockburn の use case level に従い、Sea level（user goal）を中心に書く。**
Fish level（手段）はUC図に書かず、[[03_ユースケース記述]] の基本フロー内で表現する。

| レベル                | 例                        | UC図に書くか            |
| ------------------ | ------------------------ | ------------------ |
| Cloud（戦略）          | 「会社を成長させる」               | × 抽象すぎ             |
| Kite（概要）           | 「ETラリーで高得点を獲得する」         | △ top goal として1つだけ |
| **Sea（user goal）** | **「ゲートの色と位置を把握する」**         | **◎ 中心粒度**         |
| Fish（手段）           | 「ヒントカード1を読み取る」「ログインする」   | × UC記述の基本フローへ      |
| Clam（詳細）           | 「Base64デコードする」           | × 書かない             |

ETロボコン2025 CSワークショップ P60 の **基本のユースケース図** に倣う：
> アクター → UC1: ETロボコン2025コースを攻略する → include → UC2/UC3/UC4

## 3. アクター一覧

| ID      | アクター   | 種別    | 関与するUC               | 規約出典  |
| ------- | ------ | ----- | -------------------- | ----- |
| Competitor | 競技者（人） | 主アクター | UC-1（通じて UC-3〜4 を実現） | §5.1  |

**走行体（HackSPi）と無線通信デバイス（操作台PC）はシステム内部要素** として扱い、アクターには含めない。両者の連携は **UC-3 / UC-4 の基本フロー内** で明示する（規約 §3-6 補足情報 システム分析: 走行体のみのモデルは評価が低くなる、への対応）。

> **注記:** 「開発審査員」は規約に定義されていない造語。モデル審査は要求・分析・設計・制御の各段階で「審査委員会」（参加規約 §2）により実施されるが、競技フェーズのUC図では扱わない。

## 8. 強調すべきポイント

- **UC-3 / UC-4 の基本フロー（[[03_ユースケース記述]]）に「走行体↔無線通信デバイス通信」を明記**: 規約 §3-6 補足情報の「2システム連携必須」への対応
- **アクターは競技フェーズの実行者に限定**: 競技者（競技操作者）と審判（結果判定者）。モデル審査は別フェーズであり UC図には含めない
- **手段レベルのUCを書かない**: 「ヒントカード1を読み取る」「ライン上を走行する」等の Fish level は UC図に出現させない（記述側へ）

---

## 9. 描画上の注意点（人間向け）

- **方向**: UC-3 → UC-4 の時系列順に配置
- **アクター配置**: 競技者左（主アクター）、審判右（副アクター・結果判定）
- **システム境界の矩形ラベルを必ず付ける**: 「ETラリー攻略システム」
- **印刷時のフォントは10pt以上**（規約 §クラス共通 注意事項）
- **「浮いているUC」を作らない**: 全UCが何らかのアクターか他UCに繋がっていること
- **ダイアグラムヘッダー**: `uc: [package] ETラリー攻略システム [ETラリー攻略のユースケース]`
- **凡例は不要**（UML標準記法のみで構成）
