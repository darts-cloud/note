# CS大会提出物の構成と要求モデルの位置づけ

ETロボコン2026 アプライドクラスのモデル審査（CS大会・A3横6枚）の **全体構成** と、本フォルダ（要求モデル）の役割を整理する。

> 関連: [[00_要求モデル構成]]
> 規約: `Reference/ETRC2026_dev_shinsa_rules_V1.0.pdf` §3-2, §3-5, §3-7

---

## 0. ドキュメント方針（重要）

- このフォルダは **「要求モデルに何を載せるか」の構成（中身の仕様）に集中する**
- **UML/SysML図そのものは人間が任意のツール（Cacoo / Draw.io / Astah / OmniGraffle 等）で描画する**
- マークダウンファイルでは **PlantUMLなどによる自動生成は行わない**
- 各図に対応する `.md` ファイル（[[05_SysML要求図]] / [[02_UMLユースケース図]]）は **「描くべき要素・関係性の一覧」** を表で持つ

## 3. 1ページに載せる図と表（最終提出物の構成案）

A3横 1枚に詰め込む構成。

```
┌──────────────────────────────────────────────────────────────────┐
│ タイトル: SOROT2026 要求モデル（ETラリー）       チーム名 / 所属             │
├──────────────────────────────────┬──────────────────────────────┤
│ A. SysML要求図（5層構造）              │ C. 品質要求テーブル                  │
│   [[05_SysML要求図]] の構成に従い人が描画    │   [[06_品質特性]] 抜粋          │
│   - 目標 → ステークホルダー → 機能 → 性能  │   信頼性 / 効率性 / 保守性 / 移植性     │
│   - 派生・包含・詳細化の関係線で追跡可能性     │   各 QA-xxx に指標+目標値              │
├──────────────────────────────────┼──────────────────────────────┤
│ B. ユースケース図                       │ D. 制約一覧                       │
│   [[02_UMLユースケース図]] の構成に従い人が描画 │   [[08_制約]] 抜粋             │
│   走行体+無線通信デバイス連携を強調            │   C-1xx 規約 / C-2xx 物理 / C-3xx 開発   │
│                                  ├──────────────────────────────┤
│                                  │ E. FMEA要約 (上位5件)               │
│                                  │   [[04_FMEA]] 抜粋             │
│                                  │   RPN ≥ 100 の主要故障モード          │
└──────────────────────────────────┴──────────────────────────────┘
```

**配置のコツ:**
- A（要求図）を最大面積で配置 → トレーサビリティが審査基準上最重要
- B（UC図）は走行体↔無線通信デバイスの連携が「要求段階から」見える形に
- C/D/E は表形式で密度を上げる。E（FMEA）は信頼性指標の根拠補強として右下に小さく配置
- E は地区大会版（3枚提出）でも入れる方針 → 信頼性検討の深さをアピール

## 4. 図・表の選定理由

### 採用（要求モデルで使う）

| 図/表 | 種別 | 採用理由 |
|---|---|---|
| **SysML要求図** | SysML | 規約 §3-6 で「要求図」が明示的に推奨。階層とトレースが一目で示せる |
| **UMLユースケース図** | UML | 規約 §3-2 表で「主に使用する図」筆頭。複数システム連携をアクター/UC関係で表現可能 |
| **FMEA表** | 自然言語/表 | 規約 §3-6「品質特性に指標を定義し、達成根拠を示す」への対応。信頼性 QA-REL-* の説得力強化 |
| 表（品質要求・制約） | 自然言語 | 規約 §3-6「記法は問わない」。数値指標は表が最も読みやすい |

### 後段モデルで採用（要求モデルでは使わない）

| 図 | 後段モデル | 理由 |
|---|---|---|
| BDD / IBD | システム分析 | 物理アーキ・I/F定義の中核 |
| クラス図 / オブジェクト図 | 設計 | 内部構造の詳細化 |
| ステートマシン図 | 設計 / 制御 | 振舞いの表現（設計では必須） |
| シーケンス図 / コミュニケーション図 | システム分析 / 設計 | サブシステム間振舞い |
| アクティビティ図 / ブロック線図 | 制御 | 制御戦略 |

### 採用しない

| 図 | 理由 |
|---|---|
| GSN | 規約 §3-2 で「要求図」と並列に列挙されているが、SysML要求図と冗長になるため不採用 |
| パラメトリック図 | 性能制約の数式表現は表で代替可能 |

## 6. モデル審査基準のセルフチェック

提出前に以下を満たすか確認する（規約 §3-4 より、審査委員会の視点）。

- [ ] 目標→機能要求→性能要求への **段階的分解** が追跡できるか
- [ ] 品質が **信頼性・効率性・保守性・移植性** など複数側面から検討されているか
- [ ] 各品質要求に **指標** が定義され、目標達成判定が客観的にできるか
- [ ] **FMEA** で信頼性目標の達成根拠が示されているか
- [ ] **制約** が明示されているか（特に C-308 複数システム連携の必須性）
- [ ] 走行体と無線通信デバイスの **分担・連携** が要求段階から見えるか
- [ ] 要求モデル → システム分析モデル へ **双方向トレース可能** な構造になっているか
- [ ] アブストラクトページとの整合が取れているか
- [ ] 図（要求図・UC図）は人間が描画した最終版になっているか
