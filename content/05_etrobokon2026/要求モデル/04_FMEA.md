# FMEA — ETラリー攻略システムの故障モード影響解析

要求モデル内で **信頼性要求 (QA-REL-*)** を補強する分析。
規約 §3-6 補足情報（要求モデル①）「品質特性に指標を定義し、それにより目標が達成できることが示されるとさらに良い」への対応として位置づける。

> 関連: [[00_要求モデル構成]] / [[06_品質特性]] (QA-REL-*) / [[08_制約]] / [[05_SysML要求図]]

---

## 1. 位置づけ

| 項目 | 内容 |
|---|---|
| 役割 | 信頼性品質 QA-REL-* の根拠補強 |
| 対象 | ETラリー攻略システムの主要機能（REQ-401〜409） |
| 評価軸 | 重大度 S × 発生度 O × 検出度 D = RPN |
| 要対策ライン | RPN ≥ 100（運用後に調整） |
| 提出形態 | A3横1ページの右下に **要約表**（上位3〜5件）を配置、全件は本ファイル参照 |

## 2. 評価尺度

### 重大度 S (Severity)

| S | 競技文脈での影響 |
|---|---|
| 1〜2 | ほぼ影響なし |
| 3〜4 | 軽微（数秒のロス） |
| 5〜6 | 中（1周回失敗・走行ポイント10pt減等） |
| 7〜8 | 大（リタイア相当・課題ポイント全損） |
| 9〜10 | 致命的（失格・走行体破損・フライング-35pt） |

### 発生度 O (Occurrence)

| O | 意味 |
|---|---|
| 1〜2 | ほぼ起きない（試走でも遭遇しない） |
| 3〜4 | 稀（10ターンに1回未満） |
| 5〜6 | 時々（5ターンに1回程度） |
| 7〜8 | 頻発（毎ターン発生） |
| 9〜10 | ほぼ毎回 |

### 検出度 D (Detection)

| D | 意味 |
|---|---|
| 1〜2 | テストで必ず検出 |
| 3〜4 | 高確率で検出 |
| 5〜6 | 検出可能だが見落としあり |
| 7〜8 | 検出困難（本番のみで再現） |
| 9〜10 | ほぼ検出不可 |

`RPN = S × O × D`（1〜1000）。

## 5. 対策方針（テンプレ）

各 FM に対して制御モデルで以下を設計する（要素技術の検討項目）。
このフォルダ（要求モデル）の段階では **故障モードの特定と RPN 評価** までを行う。

- 検出機構: 異常をどう検知するか
- 復帰戦略: 検出後にどう走行を継続するか
- 予防策: 設計段階で防げる工夫
- 検証方法: シミュ・実機でどう確認するか

---

## 6. 記入ガイド

- **対象機能** は必ず `REQ-xxx` を引用する（トレーサビリティ確保）
- **故障モード** は「○○が△△する」形式で具体的に
- **S/O/D** は仮置きで始めて、試走データで更新（試走前は感覚値でOK）
- **新規追加した FM** は本ファイルの表に追加し、関連列に既存 QA-REL-* / 制約 / REQ を必ず紐付け
- 数値の変更履歴は git 履歴で追跡（特別な変更ログは作らない）

## 7. レビューサイクル

- **月1回**: 試走データ反映 → S/O/D 更新 → RPN 再評価
- **対策実施後**: 該当 FM の D を再評価
- **大会開催地確定時**: C-203/C-204/C-205 関連 FM の O を見直し
