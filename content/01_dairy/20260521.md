## GitHub Projectリスケ フロー（設計中）

### 基本方針
- AIが提案 → 人間が確認 → OKなら次へ（全部AIに任せない）
- 作業を小分けにして各ステップで確認を挟む

### ステップ設計（案）
1. **現状把握**：各タスクの遅れ・前倒しを一覧化 → 合ってるか確認
2. **制約条件の整理**：動かせない締め切り・マイルストーンを特定 → 漏れ確認
3. **リスケ案の生成**：AIが日程を再配置して提案 → 1件ずつ承認/却下
4. **差分確認**：変更前後の比較表を提示 → OKならProjectに反映

### 次のアクション
- **AIでの編集と確認手法の組み立てが先決**
  - バックアップ取得スクリプト
  - リスケ案の提示フォーマット
  - 承認/却下の仕組み（差分レビュー）

## GitHub Projectリスケ計画

- AIでGitHub Projectを自動リスケしたい
- ミスに備えてリスケ前にバックアップを取る方針
- GitHub標準機能にはバックアップ・復元機能なし（CSVエクスポートのみ、インポートは非対応）
- 対象は約150件 → スクリプトで自動化する
- バックアップ時にアイテムIDも保存すれば完全復元可能（日付・ステータス・カスタムフィールド）
- 進め方：①バックアップ → ②AIが提案（GitHubは変更なし） → ③人間レビュー → ④API適用
- 次のアクション：`gh --version` で gh CLI確認から始める

## GitHub Project 残作業の定量化

- 残作業の計算式：`(終了日 - 開始日) × (100 - 進捗率) / 100`
- 要員（担当者）ごとに積算したい
- Projectに必要なフィールドは全て存在確認済み：開始日・終了日・進捗率（数値）・担当者
- 出力形式は別途検討
- 次のアクション：スクリプト設計→出力形式の相談
