## NotebookLM運用

- [x] 全員無料版のGoogleアカウントで運用する（コスト0円）
- [x] 管理者がノートブックを作成し、各担当を「編集者」として招待する
- [x] 公式ノートブックを2本用意する

  | ノートブック名 | インプット | 用途 |
  |---|---|---|
  | `仕様整理_2025` | ソースコード一式 | 新規参入者が仕様を確認・質問する |
  | `モデル採点基準` | 競技規約PDF・過去の高得点モデルPDF | UML設計のセルフチェック・採点 |

- [x] 各担当は担当モジュールのソースを追加してよい（追加・削除時は管理者に一報）
- [x] ノートブックの構成変更は管理者のみ
- [x] AI採点の結果は参考扱い（最終判断は人が行う）
- [x] 週1回、固定担当者がソースを更新する（更新後はTeamsで周知）
- [ ] NotebookLM更新の固定担当者を選定する
- [x] 自動化は将来課題（NotebookLM APIの公開状況を注視）

  **運用上の注意点**
  - ソースに使うファイルは専用フォルダで管理し削除・移動しない
  - ファイルアップロードは各自のGドライブ15GBを消費するため大容量ファイルに注意
  - メンバー離脱時は速やかにノートブックのアクセス権を剥奪する
  - 将来的な仕様変更で無料ユーザーの質問回数が制限されるリスクに注視する

## 方針決定の改善

- [x] 問題の根本（判断軸がない・情報が足りない・効果が遅い）をチームで認識統一する
- [x] 定例MTG冒頭5分をGitHub Project進捗確認の時間に固定する
  - 今週の予定進捗率 vs 実際の進捗率の乖離を全員で確認
  - 乖離が出たらその場で原因共有・対策を決める
- [x] Teams に詰まり報告チャンネルを設ける（リーダーが翌営業日中に反応する）
- [x] 方針決定時に「方針・仮説・確認指標」の3点をTeamsに記録する

  | 項目 | 内容 | 例 |
  |---|---|---|
  | 方針 | 何をやるか | タスクをすべて1〜2日単位に分解する |
  | 仮説 | いつ何が改善するか | 2週間後に詰まり報告が減る |
  | 確認指標 | 何で判断するか | Teams詰まり報告の件数 / GitHub進捗率 |

- [x] 定例MTGで「前回の方針は仮説通りか」を1分確認する
- [x] 仮説が外れた場合は「なぜ外れたか」を記録して次の方針に活かす
