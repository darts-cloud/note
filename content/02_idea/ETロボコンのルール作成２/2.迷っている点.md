# 2.迷っている点

## 1. 記録場所の選定
- 方針・仮説・結果の記録をどこに残すべきか。
    - **案A: Teams**（フロー型）：手軽だが、過去の知見を振り返りにくい。
    - **案B: GitHub Discussions/Issues**（ストック型）：コードと紐付くが、入力の心理的ハードルがやや高い。
    - **案C: Markdownファイル（このリポジトリなど）**：構成が自由だが、共有と通知が弱い。
- 回答：案B

## 2. ペア制の「強制力」
- どこまでルール化すべきか。
    - **案A: 完全に自由**（アドバイスが必要な時だけ）：機能しない可能性が高い。
    - **案B: 作業時間を固定してペアワーク**：進捗は出るが、個人の作業時間が削られる不満が出るかも。
    - **案C: コードレビューのペアを固定するのみ**：負担は少ないが、設計段階での「詰まり」は解消しにくい。
    
- 回答：まずPM定例会の題材とする必要あり

## 3. 「詰まり」報告の閾値
- 「何をもって詰まったとみなすか」の定義。
    - 時間（例：2時間）で切るか。
    - 感情（例：イライラし始めたら）で切るか。
    - 作業内容（例：環境構築で1時間以上）で切るか。
    
- 回答：これは保留

## 4. NotebookLMへの「ドキュメント」投入範囲
- 仕様書、設計書、ソースコードのどこまでを入れるべきか。
- 全部入れるとAIが混乱する（矛盾する情報がある場合）リスク。
- 「常にソースコードを正とする」という優先順位をAIに徹底させる方法。

- 回答：ドキュメントは設計書と規約を想定

## 5. モチベーション維持とルールのバランス
- ルールを増やしすぎることによる「管理されている感」の増大。
- どこまでを「ルール」とし、どこまでを「推奨（ベストプラクティス）」とするか。
- リーダーが「ルールを守らせる人」になりすぎて、メンバーとの距離が開くリスク。

- 回答：良い塩梅の案が欲しい。
